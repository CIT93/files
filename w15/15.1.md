# Why return new Error vs error:

It's important to be precise about **why** you use `new Error()` with `reject()` in a Promise and how it relates to `try...catch`.

You **can** still use a `try...catch` without using `new Error()`, but you'll lose crucial debugging information and violate a best practice.

Here's the breakdown:

-----

## 🛑 Why `new Error()` is Essential for Promises

### 1\. The Role of `reject()` and `catch`

When you `reject(value)` a Promise, you are just passing the `value` to the `.catch()` handler (or the `catch` block of an `async` function).

  * **`reject(someValue)`** → The `.catch()` block receives `someValue`.

### 2\. If You Reject a String or Number

If you simply wrote `reject(error.message)` (which would be a string), the `catch` block would receive a string.

```javascript
// Example of bad practice
reject("Permission denied."); 

// Later in an async function:
try {
    await somePromise;
} catch (e) {
    console.log(e); // e is just the string "Permission denied."
    // console.log(e.stack); // 💥 Fails! The string has no .stack property.
}

The issue is that the error object is missing its **stack trace** (`.stack`), which is the most valuable part of an error during debugging, as it tells you the exact line of code that triggered the failure.


**In short: You can still use `try...catch` regardless of what you pass to `reject()`. However, you should use `new Error()` with `reject()` to ensure you pass a proper `Error` object that contains the essential debugging `stack` property.**
